# Wall

## Architectural Notes

- Phase 0.1: The workspace currently contains only `SUMMARY.md`, `Allhandoff.md`, and `.env` - implementation must start from a blank project skeleton - no existing codebase patterns are available to inherit.
- Phase 0.1: Required runtime integrations are already implied by provided environment variables for Telegram, Groq, and OpenRouter/DeepSeek - configuration should consume existing env keys before introducing any new ones - avoid unnecessary config surface.
- Phase 0.2: Execution will follow the external protocol in `Allhandoff.md` rather than batching work - each reply after planning must complete one atomic task with verification before advancing - task granularity must stay strict.
- Phase 1.2: The SQLite `tasks` schema now defines typed fields for user ownership, task description, scheduled time, status, and timestamps - later database tool wrappers must reuse this single source of truth - avoid parallel task DTO definitions.
- Phase 1.2: The TypeScript baseline remains CommonJS, so `verbatimModuleSyntax` was disabled and TS6 deprecation silenced in `tsconfig.json` - standard module imports otherwise fail compilation - preserve this compiler contract until an explicit ESM migration is planned.
- Phase 1.3: SQLite schema initialization is performed by an explicit `initDatabase()` bootstrap instead of introducing migrations this early - the project needs deterministic local table creation before higher-level task tools exist - keep database setup minimal until migration requirements appear.
- Phase 2.1: Database operations are exposed as async wrappers with colocated JSON schemas in `src/db/tools.ts` - the later OpenAI tool layer needs typed callable contracts without adding schema dependencies yet - keep tool definitions close to the Drizzle task model until orchestration code exists.
- Phase 2.2: Wrapper verification now lives in a dedicated compiled script `src/db/verify-tools.ts` - task tool behavior must be reproducible without ad hoc REPL commands - keep future database checks scriptable through npm entrypoints.
- Phase 3.1: Mem0 is configured via `mem0ai/oss` with local history/vector SQLite files because the provided `.env` has no Mem0 platform key - long-term memory must stay local and reuse existing OpenRouter credentials instead of inventing new secrets.
- Phase 3.1: OpenRouter is the single OpenAI-compatible endpoint for both chat and embeddings, with `deepseek/deepseek-v4-flash` for responses and `openai/text-embedding-3-small` for memory vectors - future orchestration should keep model routing centralized in `src/ai/openrouter.ts`.
- Phase 3.2: Runtime env loading now supports the repository's existing `key:value` `.env` format in addition to standard dotenv parsing - provided secrets must be reused as-is without forcing a file rewrite - future runtime entrypoints should rely on `src/config/env.ts` rather than parsing env files independently.
- Phase 3.2: Mem0 OSS may normalize inserted text into a distilled fact during storage and retrieval - verification and later orchestration should treat memory as semantic context rather than exact string echo - assertions should check durable meaning, not byte-for-byte round-trips.
- Phase 4.1: Telegram intake uses direct Bot API long polling and file downloads while Groq transcription reuses the existing OpenAI SDK against Groq's compatible endpoint - this avoids adding bot-specific dependencies before orchestration exists - keep message intake transport thin and isolated under `src/telegram/`.
- Phase 4.2: Short-term conversation context is an in-memory per-user sliding window capped at 12 messages inside `src/telegram/history.ts` - orchestration needs recent local context before long-term memory/tool calls exist - keep this layer process-local and dependency-free until runtime persistence is explicitly required.
- Phase 4.3: Voice-path verification now runs through a compiled dependency-injected harness in `src/telegram/verify-voice.ts` - transcript logging must stay reproducible without live Telegram/Groq calls - preserve the default runtime behavior while keeping handlers injectable for future integration checks.
- Phase 5.1: The assistant runtime now uses a dedicated OpenRouter tool-calling loop in `src/ai/orchestrator.ts` that injects short-term history, retrieved Mem0 context, and SQLite task tools into one reply pipeline - Telegram text and voice handlers should stay transport-thin and delegate all business orchestration into this single layer.
- Phase 5.1: Durable memory writes are triggered asynchronously after a successful assistant reply while memory retrieval failures degrade gracefully - Telegram responses should not block on Mem0 persistence or fail outright when long-term memory is temporarily unavailable.
- Phase 5.2: Tool schemas exposed to the LLM must not require transport-owned identifiers such as Telegram `userId` - the first end-to-end text verification failed because the model asked the user for an internal ID instead of calling `add_task` - keep user scoping injected server-side inside the orchestrator/tool executor.
